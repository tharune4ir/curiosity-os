## Translation model and backend coverage design

**Part 2 ÔÇö Translation model (node engine Ôćĺ teacher-facing activity database)**

### Why one-node-to-one-activity is the wrong public model
The backend doctrine was built for **node-first teaching** (1 node = 1 lesson + assignment + reflection + tracking) in V1. ţłÇfileciteţłéturn0file1ţłü  
But Curiosity OS 2.0 is explicitly being shaped as **teacher-facing and activity-first**, where the teacherÔÇÖs unit is an ÔÇťactivityÔÇŁ (run-mode, observation, adaptation). A strict 1:1 mapping would surface the backendÔÇÖs granularity as teacher burden:

- It would force teachers to choose among dozens of micro-nodes (especially in Cognition) rather than run coherent classroom experiences. ţłÇfileciteţłéturn0file0ţłü  
- It would overemphasize vocabulary labels (bias names, model names) rather than capability outcomesÔÇödirectly violating ÔÇťyour teaching is judged by what the student can now do.ÔÇŁ ţłÇfileciteţłéturn0file1ţłü  
- It would encourage graph-like navigation (ÔÇťwhich node next?ÔÇŁ) rather than activity-first planning (ÔÇťwhat do I run tomorrow, and what evidence will it generate?ÔÇŁ). ţłÇfileciteţłéturn0file1ţłü

### The correct unit of translation
**Correct translation unit:** a **Capability Mission** (teacher-facing activity) that fuses:
- one **core lens** (dominant node cluster),
- one **practice mechanic** (Sandbox-style rehearsal OR artifact production),
- one **evidence artifact** (what the teacher can observe / collect),
- one **transfer move** (where it travels next). ţłÇfileciteţłéturn0file1ţłü

In other words: **nodes are lenses; activities are missions that force lens-use and produce evidence.** This aligns with the operating doctrine that every lesson must leave evidence, and that transfer is mandatory. ţłÇfileciteţłéturn0file1ţłü

### How multiple backend nodes should combine into an activity
Use a controlled bundling pattern:

- **Bundle rule:** 3ÔÇô8 nodes per activity, with **1ÔÇô2 ÔÇťcoreÔÇŁ nodes**, 2ÔÇô4 ÔÇťsupportÔÇŁ nodes, and optional ÔÇťextensionÔÇŁ nodes.  
- **Cross-wing rule:** most flagship activities should be intentionally **hybrid**:  
  - Decode Ôćö Cognition for truth-seeking (graphs, odds, evidence hygiene). ţłÇfileciteţłéturn0file0ţłü  
  - Cognition Ôćö Relate for disagreement, peer pressure, manipulation defense. ţłÇfileciteţłéturn0file0ţłü  
  - Any wing Ôćö Sandbox for embodiment and rehearsal. ţłÇfileciteţłéturn0file0ţłü  
- **Teacher simplicity rule:** the teacher never sees ÔÇť8 nodesÔÇŁ; they see ÔÇťone activity with a clear promise, steps, and watch-fors.ÔÇŁ The node mapping is internal-only. ţłÇfileciteţłéturn0file1ţłü

### How hidden internal mappings should work
Internally, treat each activity like a ÔÇťconstellationÔÇŁ with weights (not just tags):

- `core_node_hooks` (weighted 0.6ÔÇô1.0): the activity cannot exist without these nodes.  
- `support_node_hooks` (weighted 0.2ÔÇô0.6): reinforced or invoked during the activity.  
- `embodiment_hooks` (Sandbox practice atoms / outcomes): how the activity converts concept Ôćĺ behavior. ţłÇfileciteţłéturn0file0ţłü  
- `evidence_dimensions_targeted`: which of Generate/Notice/Represent/Reason/Transfer/Reflect should be most observable. ţłÇfileciteţłéturn0file1ţłü

This enables recommendation without graph UI: the app can suggest ÔÇťnext best activitiesÔÇŁ based on weak evidence dimensions, not on node adjacency screens. ţłÇfileciteţłéturn0file1ţłü

### Proposed translation framework (concrete)
**Backend nodes Ôćĺ Capability clusters Ôćĺ Flagship activity patterns Ôćĺ Teacher-facing collections**

1) **Backend nodes** (147) remain the deepest truth. ţłÇfileciteţłéturn0file0ţłü
